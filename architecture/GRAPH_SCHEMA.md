# Knowledge Graph Node Schema

This document details the metadata structure, indexing tags, and namespace placeholders defined for every `KnowledgeNode` in the CFA Operating System.

---

## 1. Primary Schema Properties

Every Node, regardless of its category, exposes a standard set of read-only properties:

* **`id`**: Unique string token (persistent UUID for syllabus/formulas/resources, unique key for notes/sessions).
* **`type`**: Node classification string (`Subject` | `Reading` | `LOS` | `Formula` | `Study Note` | `Resource` | `Study Session`).
* **`title`**: Human-readable name.
* **`description`**: Core textual payload (abstract summaries, markdown outlines, or logs).
* **`createdDate`**: ISO timestamp.
* **`modifiedDate`**: ISO timestamp.
* **`tags`**: Indexing tags (e.g. `'subject'`, `'reading'`, `'los'`, `'equation'`).
* **`difficulty`**: Level III syllabus rating (`Easy` | `Medium` | `Hard` | `Unspecified`).
* **`confidence`**: Candidate qualitative confidence score (`1` to `5` or `null`).
* **`parentIds`**: References establishing strict structural parents (e.g., Reading ID parent of an LOS).
* **`childIds`**: References to children (e.g., Reading node child IDs pointing to LOS items).
* **`relatedIds`**: Cross-curriculum links (e.g., related LOS pairs).
* **`referenceCount`**: Monotonic count of incoming relationship edges.
* **`relationshipCount`**: Total node degree (incoming + outgoing).
* **`searchWeight`**: Float multiplier boosting title match queries (e.g., Subject = 1.5, Session = 0.8).

---

## 2. Nested Metadata Namespaces

To prevent flat-property bloat and prepare the system for subsequent AI, search, and NotebookLM sync engines, all advanced metadata is structured under the `metadata` namespace:

### A. Search Namespace (`metadata.search`)
Normalized tokens for offline/local keyword matching:
* `normalizedTitle`: Lowercased, trimmed representation of the node title.
* `keywords`: Array of individual keywords extracted from the title.
* `aliases`: Alternative terms (e.g., "Singer Terhaar", "Singer-Terhaar Model", "Singer Model").
* `searchTokens`: Complete query token list combining keywords, aliases, and lowercase tags.
* `embeddingHash` (Future): Hash representing local vector state.
* `semanticScore` (Future): Query relevance rating.

### B. NotebookLM Namespace (`metadata.notebookLM`)
Placeholders for subsequent Vector RAG indexing:
* `notebookLMChunkIds`: Chunk identifiers generated by document parsers.
* `embeddingIds`: Matching vector embedding IDs in the client/server database.
* `sourcePdfIds`: Absolute file paths to source PDFs.
* `chunkReferences`: Array of exact text snippets matched to the node.
* `vectorIds`: Matching semantic vector store IDs.
* `groundingSources`: Direct source bibliography anchors.

### C. AI Namespace (`metadata.ai`)
Placeholders for automated tutor prompt synthesis:
* `aiSummary`: AI-generated capsule summary.
* `aiExplanation`: AI-generated conceptual breakdown.
* `aiQuiz`: Collection of mock multiple-choice questions grounded in the node content.
* `aiEssayPrompt`: Tailored essay test prompts.
* `aiMisconceptions`: Frequently confused ideas flagged by tutor logs.
* `aiConfidenceEstimate`: AI-assessed confidence (0.0 to 100.0) based on practice scores.
* `aiTutorNotes`: Diagnostic notes compiled by the virtual coach.
